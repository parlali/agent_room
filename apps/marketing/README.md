# Marketing

Public marketing site for Agent Room.

## Development

```bash
bun run marketing:dev
```

The dev server runs on port `3100` and serves:

- React routes for `/`, `/features`, `/pricing`, `/security`, `/source`, `/terms`, and `/privacy`
- Product screenshots from `public/assets/marketing/`
- Generated OG share image from `packages/brand/exports/marketing/` after `brand:export:marketing`
- Waitlist submissions at `POST /api/waitlist`

## Build and preview

```bash
bun run brand:export:marketing
bun run marketing:build
bun run marketing:preview
```

`marketing:preview` serves `dist/` with the waitlist API on port `4173` by default.

## Cloudflare Workers

This repo includes a Wrangler config for deploying the marketing site as a Cloudflare Worker with Static Assets and D1. It is useful for maintainers and forks that want a single Cloudflare deployment for the static site and waitlist API.

The root `wrangler.jsonc` defines:

- `apps/marketing/worker.ts` as the Worker entrypoint
- `apps/marketing/dist` as the static assets directory
- `WAITLIST_DB` as the D1 binding
- `apps/marketing/db/migrations` as the D1 migrations directory

For Git-connected Cloudflare builds from the repo root, use:

| Setting        | Value                                                       |
| -------------- | ----------------------------------------------------------- |
| Build command  | `bun run brand:export:marketing && bun run marketing:build` |
| Deploy command | `bun run marketing:deploy`                                  |

The deploy script runs `wrangler deploy` and then applies the remote D1 migrations. Wrangler reads the static asset path, Worker entrypoint, D1 binding, and SPA fallback behavior from `wrangler.jsonc`. Forks can rename the Worker and D1 database in `wrangler.jsonc` if they want different resource names in their Cloudflare account.

Validate the Worker bundle locally without deploying:

```bash
bunx wrangler deploy --dry-run
```

## Environment variables

| Variable                        | Default                                | Purpose                                          |
| ------------------------------- | -------------------------------------- | ------------------------------------------------ |
| `VITE_ALLOWED_HOSTS`            | none                                   | Extra dev-server hostnames, comma-separated      |
| `VITE_WAITLIST_API_URL`         | `/api/waitlist`                        | Waitlist POST endpoint for the browser form      |
| `MARKETING_WAITLIST_DB`         | `apps/marketing/.data/waitlist.sqlite` | SQLite database for waitlist submissions         |
| `MARKETING_WAITLIST_RATE_LIMIT` | `8`                                    | Max submissions per client IP per hour           |
| `PORT`                          | `4173`                                 | Port for `marketing:preview` / `marketing:start` |

## Marketing assets

### Product screenshots

Committed in `apps/marketing/public/assets/marketing/`. Filenames are referenced in `apps/marketing/src/content/assets.ts`. Vite copies them to `dist/assets/marketing/` at build time.

- `hero-desktop-room-console.png` — Home hero (desktop)
- `hero-mobile-room-console.png` — Home hero (mobile)
- `capabilities-artifacts-dashboard.png` — Features page
- `security-audit-runtime.png` — Security page

Replace files in `public/assets/marketing/` when updating captures.

### OG share image

Generated by the brand package and not committed:

```bash
bun run brand:export:marketing
```

Writes `packages/brand/exports/marketing/og-share.png`. The marketing Vite build emits it to `dist/assets/marketing/og-share.png` through `@agent-room/brand/vite`. Docker runs the export step before build.

## Waitlist storage

Local development and preview store submissions in SQLite, not email. Inspect local records with:

```bash
sqlite3 apps/marketing/.data/waitlist.sqlite "select id, created_at, email, company, interest from waitlist_submissions order by id desc limit 20;"
```

Cloudflare deployments store submissions in D1 through the `WAITLIST_DB` binding.

The form includes a honeypot field and server-side rate limiting. Bots that fill the honeypot receive a success response without persisting data.

## Docker

The root `Dockerfile` builds the marketing site into `apps/marketing/dist/` alongside the self-hosted app. Marketing static assets and favicons are included in that build output.
